## Contexto

Existem dois paths com o nome `intelagent` no monorepo que compartilham a mesma estrutura de workspace Cargo mas são **projetos fundamentalmente diferentes**.

---

### Projeto A — `phantom/intelagent/`

**Filosofia**: "Replace prompts and hope with specifications and proofs."

**O que é**: Framework abstrato de orquestração de agentes com ambições de plataforma.

**Escopo**: ZK proofs (Circom), DAO governance (Algorand), `$INTEL` token, compliance GDPR/HIPAA/SOC2, peer review protocol, quality gates formais.

**Estado de implementação**:

| Crate | Status |
|-------|--------|
| `core` | Traits implementados: `Agent`, `Task`, `QualityGate`, `Context`, `Proof` — com `Debug + Clone + clone_box()` correto |
| `soc` | Binário GTK4 Phase 0 — UI scaffolding, CommandPalette + LogViewer |
| `security`, `governance`, `memory`, `quality`, `mcp`, `cli` | Scaffolding — módulos declarados, sem implementação |

**Identidade declarada**: `Cargo.toml` aponta para `repository = "https://github.com/kernelcore/intelagent"` — intenção standalone explícita.

**Licença**: Apache-2.0.

---

### Projeto B — `phantom-ray/phantom-stack/libs/intelagent/crates/soc/`

**Filosofia**: "Observability, not abstractions. Kernel puro."

**O que é**: SOC kernel concreto para orquestração de agentes phantom dentro do phantom-stack.

**Escopo**: Processar documentos via phantom binary local (`tokio::process::Command`), observabilidade total de tasks e agentes.

**Estado de implementação**:

| Componente | Status |
|------------|--------|
| `TaskQueue` | ✅ Implementado — MPSC unbounded, AtomicU64 stats, `Task::ProcessDocument` |
| `AgentPool` | ✅ Implementado — lifecycle Spawn/Busy/Idle/Kill, RwLock<HashMap> |
| `EventBus` | ✅ Implementado — broadcast channel, 1000 event buffer, full telemetry |
| `Metrics` | ✅ Implementado — lock-free atomics, avg/min/max timing, success rate |
| `Scheduler` | ✅ Implementado — round-robin task routing, PhantomAgent integration |
| `PhantomAgent` | ✅ Implementado — chama `phantom` binary via PATH, `nix develop` aware |
| GTK4 UI | ✅ Phase 0 — app_id `dev.voidnix.intelagent.soc`, dark theme, SOCLayout |

**Status declarado no KERNEL.md**: `✅ BACKEND KERNEL COMPLETO. Compilando limpo. Pronto para integração com UI de observabilidade.`

**Integração**: Está corretamente posicionado como lib dentro de `phantom-ray/phantom-stack/libs/` — serve o phantom-stack, não o umbrella geral.

**Licença**: MIT (MANIFEST).

---

### Por que a confusão é perigosa

A estrutura superficial de Cargo workspace é idêntica nos dois. Quem vê os dois paths side-by-side assume que um é cópia desatualizada do outro — e a decisão natural é deletar o de `phantom-ray`. Isso seria um erro: descartaria o único componente funcional dos dois.

Diferenças críticas que revelam a distinção real:

| Dimensão | Projeto A (`phantom/`) | Projeto B (`phantom-ray/`) |
|----------|------------------------|---------------------------|
| Filosofia | Especificações + provas formais | Observabilidade + kernel direto |
| Abstração | Alta — traits genéricos `Agent<T>` | Baixa — `PhantomAgent` concreto |
| Implementação | Scaffolding (1 de 8 crates real) | Funcional (kernel completo) |
| Acoplamento | Agnóstico (quer 10+ projetos) | Acoplado a phantom binary via PATH |
| Governança | DAO on-chain (Algorand) | Nenhuma — observabilidade local |
| SOC target | Framework para múltiplos SOCs | SOC específico do phantom-stack |

---

## Decisão

**Tratar como dois projetos distintos. Ação em cada um:**

### Projeto A — `phantom/intelagent/` → desacoplar como `master/intelagent/`

1. Mover para `master/intelagent/` como projeto de primeiro nível no umbrella
2. Descartar `crates/soc` — o GTK4 Phase 0 scaffolding não acrescenta ao framework; o phantom-ray SOC kernel é o executável canônico
3. Definir boundary: phantom expõe REST API como worker de Layer 0; intelagent consome via `PhantomWorker` HTTP client
4. MVP mínimo antes de qualquer release: `crates/core` funcional + `PhantomWorker` (reqwest) + 1 integration test end-to-end
5. Não implementar ZK, DAO, token antes do MVP estar operacional

### Projeto B — `phantom-ray/.../libs/intelagent/` → renomear, manter em phantom-ray

1. Renomear para `phantom-ray/phantom-stack/libs/phantom-soc-kernel/` (ou `phantom-ops-kernel/`)
2. Manter posicionamento como lib interna do phantom-stack — está correto onde está
3. Registrar no ROADMAP.md como ativo funcional (atualmente invisível na status matrix)
4. Avaliar se deve ser o backend do `phantom-ray/phantom-soc/control-plane` (que hoje usa EventBus independente) — potencial unificação

### Boundary pós-decisão

```
master/intelagent/          (framework abstrato, standalone)
    └── PhantomWorker → HTTP → phantom-api :8008
    └── CerebroWorker → HTTP → (futuro)

phantom-ray/phantom-stack/
    └── libs/phantom-soc-kernel/   (SOC kernel concreto, funcional)
            └── PhantomAgent → binary → phantom CLI
    └── phantom-soc/control-plane  (GTK4 SOC UI — pode usar o kernel acima)
```
