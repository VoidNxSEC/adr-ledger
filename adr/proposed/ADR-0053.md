## Context

The initial version of the browser extensions article (v1) covered the technical attack surface, documented real-world incidents (DarkSpectre, Trust Wallet, GlassWorm, Cyberhaven), and mapped enterprise controls — runtime host restrictions, version pinning, detection architecture, and extension lifecycle management.

The article was technically complete. It addressed engineers who build the controls and managers who approve the policies. But it treated user behavior as an externality — something that exists outside the system being described.

This is the same mistake the article criticizes in enterprise security teams: addressing the tooling while ignoring the operating conditions. The permission model, the shadow IT growth, the incentive misalignment between user utility and organizational risk — these are not peripheral concerns. They are the reason most enterprise extension policies fail in practice, regardless of technical correctness.

The gap became clear when reviewing the v1 draft: the article argued that "the allowlist is the floor, not the ceiling" but never explained *why* the floor is insufficient from the user's perspective. The conclusion landed as an engineering opinion rather than a systemic observation.

## Decision

Blog content that covers security topics must treat user behavior as part of the threat model — not as an externality that exists outside the system being analyzed.

Specifically:

1. **User behavior as attack surface**: When discussing security topics, user interaction patterns (permission fatigue, invisible infrastructure, shadow IT, incentive misalignment) must be treated as part of the threat model, not as a footnote. The audience remains engineers and managers — but both make better decisions when they see the human layer of the system they're operating.

2. **Operational completeness over compliance completeness**: Content scope is not bounded by regulatory frameworks. A control that is technically correct but operationally ignored is a failed control. The content must explain *why* controls fail in practice, which requires understanding user behavior.

3. **Systemic framing**: User behavior problems must be framed as systemic failures (bad incentive structures, uninformative permission prompts, lack of feedback mechanisms), never as individual negligence. The tone is analytical, not condescending.

## Rationale

### Drivers

- Technical content that ignores user behavior produces recommendations that fail at deployment — the browser extensions article demonstrated this directly
- Engineers and managers both make better decisions when they see the full picture, including the human layer of the attack surface
- Security analysis that stops at the tooling layer is incomplete — the operating conditions (how users actually behave) determine whether controls succeed or fail

### Alternatives Considered

#### Option A: Purely technical content
- **Pros:** Faster to write, clearer scope, easier to maintain technical credibility
- **Cons:** Misses the operational reality that determines whether controls work; limits audience to implementers only
- **Why rejected:** The browser extensions v1 draft proved that a technically complete article can still be incomplete. The user behavior section in v2 made the argument stronger, not weaker.

#### Option B: Compliance-first content
- **Pros:** Appeals to governance stakeholders, maps directly to audit requirements
- **Cons:** Compliance framing narrows the scope artificially — real security decisions involve cost, usability, user behavior, and organizational culture, not just regulatory checkboxes
- **Why rejected:** Compliance is a constraint, not a strategy. Content anchored only to compliance misses the engineering and user dimensions that determine real-world outcomes.

### Trade-offs

- Broader scope increases editorial effort and article length, but produces a more honest and complete analysis
- Addressing user behavior requires careful tone — must always frame as systemic analysis, not blame
- Longer articles may reduce casual engagement, but the target audience (engineers and managers) values depth over brevity

## Consequences

### Positive

- Articles present the complete operational picture, not just the tooling layer
- Threat models that include the human layer are more credible and more useful to decision-makers
- Recommendations become actionable because they account for real-world adoption barriers

### Negative

- Higher time investment per article
- Longer read times (the browser extensions article went from ~9 to ~11 minutes with the v2 additions)

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Scope creep — trying to serve every audience dilutes the core thesis | Low | Medium | Each article must have one clear primary argument; the user perspective supports it, never replaces it |
| Tone drift — user behavior sections could read as patronizing | Low | High | Frame all user behavior as rational response to bad system design, never as individual failure |
| Diminishing returns on length | Medium | Low | Target 10-12 min read time as ceiling; cut aggressively if user section doesn't strengthen the core argument |
