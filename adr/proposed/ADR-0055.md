## Context

M1 through M6 are complete. The voidnxlabs stack now has:

- **Event bus** (Spectre/NATS) with NKey auth, TLS, JetStream streams
- **Document intelligence** (Phantom) with DAG pipeline, FAISS RAG, NATS publisher
- **Knowledge extraction** (Cerebro) with NATS consumer/publisher loop
- **Zero-trust LLM proxy** (SecureLLM Bridge) with real Prometheus metrics + NATS events
- **Network SIEM** (Owasaka) publishing `network.*.v1` events
- **System monitoring** (ai-agent-os) publishing `system.metrics.v1`
- **SOC dashboard** (phantom-soc) consuming events from the bus
- **CI/CD** (sentinel) with full E2E, chaos, and performance test suites
- **ML inference bridge** (ml-ops-api) with NATS events
- **NixOS packaging** and cross-platform distribution scripts

The infrastructure is production-grade but has **zero external users and zero public visibility**.

M7 (production deploy on real hardware) and M8 (distribution — Homebrew, .deb, .rpm, winget) are the next milestones, but they require:
1. Understanding which platforms users actually run (NixOS? Docker? Ubuntu?)
2. Real-world stress testing from people other than the author
3. Community feedback to prioritize M8 packaging targets

Shipping M7+M8 in a vacuum risks building packaging nobody downloads.

## Decision

**Pause active development after M6 NKey fix (2026-03-30) and enter a go-to-market phase.**

During the pause:
- No new features, no new milestones
- Focus: public announcement, community building, README polish
- Fix the two known low-effort gaps (shared volume, untracked test file, ROADMAP matrix)
- Resume M7 on first external contributor issue OR 2026-05-01, whichever comes first

## Rationale

### Drivers

- M1–M6 complete: the stack is demonstrably real and working
- Zero public visibility = zero validation signal
- M8 packaging effort (4 platforms) is significant — should be informed by who actually wants to use it
- Sustainable pace: a clear boundary prevents continuous feature creep

### Alternatives Considered

#### Option A: Push straight to M7
- **Pros:** Maintain momentum
- **Cons:** Still no users; M8 targeting blind
- **Why rejected:** Market signal must come before packaging investment

#### Option B: Continue at current pace indefinitely
- **Pros:** More features faster
- **Cons:** Diminishing returns; burnout; still no users
- **Why rejected:** External validation is a prerequisite for correct prioritization

### Trade-offs

- Short-term velocity loss in exchange for better M8 priority signal
- Some public exposure before 100% polish (acceptable — the code is solid enough)

## Consequences

### Positive

- Community feedback informs M8 platform priority
- Early adopters stress-test before M7 production hardening
- Architecture docs + ADR ledger serve as marketing assets (they demonstrate rigor)
- Maintainer energy preserved for a focused M7 sprint

### Negative

- M7 slips by at least a few weeks
- Bug reports may arrive before capacity to address them

### Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Low community traction | medium | low | Pre-identify 3–5 specific communities before announcing |
| Security report before M3 review | low | high | Audit public repos for secrets before any announcement |
| Scope creep from requests | medium | medium | Gate M7 scope strictly on existing ROADMAP |
